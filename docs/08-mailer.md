# doido-mailer — Spec

Rails analogue: **Action Mailer**

## Decisions (resolved in interview)

- **Delivery strategies: both** — `deliver_now()` (inline async) and `deliver_later()` (enqueued via `doido-jobs`)
- **Templates: inside `views/`** — same Tera engine and resolution convention as controllers

## Template Convention

Mirrors controller view resolution:

```
views/
  mailers/
    user_mailer/
      welcome.html.tera        ← HTML part
      welcome.text.tera        ← plain-text part
      reset_password.html.tera
      reset_password.text.tera
```

Template key: `"mailers/user_mailer/welcome"` → resolves both `.html.tera` and `.text.tera` parts.  
Both parts are rendered and composed into a multipart MIME message automatically.  
If only `.html.tera` exists, a plain-text fallback is auto-generated by stripping HTML tags.

## Mailer Design

```rust
#[mailer]
struct UserMailer;

impl UserMailer {
    async fn welcome(user: &User) -> Mail {
        Mail::new()
            .to(&user.email)
            .from("noreply@myapp.com")
            .subject("Welcome to Doido!")
            .template("mailers/user_mailer/welcome", json!({ "user": user }))
    }

    async fn reset_password(user: &User, token: &str) -> Mail {
        Mail::new()
            .to(&user.email)
            .from("noreply@myapp.com")
            .subject("Reset your password")
            .template("mailers/user_mailer/reset_password", json!({ "user": user, "token": token }))
    }
}
```

## Two Delivery Strategies

```rust
// deliver_now — awaits SMTP inline, blocks until sent
UserMailer::welcome(&user).deliver_now().await?;

// deliver_later — enqueues a job via doido-jobs, returns immediately
UserMailer::welcome(&user).deliver_later().await?;
```

`deliver_later` serializes the `Mail` struct to JSON and enqueues a `MailDeliveryJob` in `doido-jobs`.  
The job worker calls `deliver_now` when it processes the job.

## `Mail` Struct

```rust
pub struct Mail {
    pub to:       Vec<String>,
    pub cc:       Vec<String>,
    pub bcc:      Vec<String>,
    pub from:     String,
    pub reply_to: Option<String>,
    pub subject:  String,
    pub html:     Option<String>,    // rendered HTML body
    pub text:     Option<String>,    // rendered plain-text body
    pub attachments: Vec<Attachment>,
}
```

## Delivery Backends (`Deliverer` trait)

```rust
pub trait Deliverer: Send + Sync {
    async fn deliver(&self, mail: &Mail) -> Result<()>;
}
```

| Backend | Feature flag | Description |
|---------|-------------|-------------|
| `SmtpDeliverer` | default | Sends via SMTP (configurable host/port/TLS) |
| `TestDeliverer` | always available | Captures sent mail in memory; never sends |
| `LogDeliverer` | default dev | Logs mail content to tracing output |

Active backend selected via config:

```toml
[mailer]
deliverer = "smtp"        # "smtp" | "log" | "test" | custom
from = "noreply@myapp.com"

[mailer.smtp]
host = "smtp.myapp.com"
port = 587
tls = true
username = "${SMTP_USER}"
password = "${SMTP_PASSWORD}"
```

In test environment, `TestDeliverer` is always used regardless of config.

## Test Helper

```rust
// In tests
use doido_mailer::testing::MailBox;

let mailbox = MailBox::new();
UserMailer::welcome(&user).deliver_now().await?;

assert_eq!(mailbox.len(), 1);
let mail = mailbox.last().unwrap();
assert_eq!(mail.to, vec!["user@example.com"]);
assert!(mail.subject.contains("Welcome"));
```

## Known Requirements

- `#[mailer]` macro wires the struct into the framework
- Each mailer method returns a `Mail` value (not yet sent)
- `Mail` is `Serialize`/`Deserialize` — needed for `deliver_later` job enqueue
- Templates resolved via `doido-view`'s `TemplateEngine` — same Tera instance
- Multipart auto-composed from `.html.tera` + `.text.tera` pair
- `deliver_later` enqueues `MailDeliveryJob` in `doido-jobs`
- `TestDeliverer` auto-active in test environment; `MailBox` injectable for assertions
